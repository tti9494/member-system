## 어떤 문제들이 있었고, 왜 수정했나요?

### 1. 관리자 인증 없음 (가장 중요)
처음 만든 API에는 누구든 `/members` 조회, `/approve`, `/blacklist` 를 호출할 수 있었어요.
URL만 알면 다른 사람을 블랙리스트에 올리거나 신청자 명단 전체를 볼 수 있었죠.
→ `X-Admin-Key` 헤더를 추가해서, 키를 모르는 사람은 접근 못하게 막았습니다.

### 2. 동의 체크박스가 무시되고 있었음
`join-full.html`에서 동의 버튼을 안 눌러도 서버에 `consent_personal: true` 로 전달되고 있었어요.
개인정보 동의 없이 데이터가 수집되는 건 법적으로 문제가 됩니다.
→ `checkbox.checked` 실제 값을 읽도록 수정했습니다.

### 3. 똑같은 코드가 두 군데 있었음 (`calculate_grade` vs `_grade_count`)
`db.py`의 `calculate_grade`와 `main.py`의 `_grade_count`가 동일한 등급 계산을 했는데,
`calculate_grade`는 존재하지 않는 키(`_can_code_provided`)를 참조해서 항상 0이 나왔어요.
실제로는 `_grade_count`만 쓰이고 있었습니다.
→ 혼란을 없애기 위해 `calculate_grade`를 완전히 제거했습니다.

### 4. "SELECT" 한 글자에 막히는 보안 필터
`security_checker`가 "SELECT 버튼을 눌렀더니..." 같은 정상 문장도 SQL 공격으로 판단했어요.
→ 실제 SQL 구문(`SELECT ... FROM`, `DROP TABLE` 등) 패턴만 감지하도록 수정했습니다.

### 5. 로그에 "locked" vs "unlocked" 혼동
잠금 해제 처리인데 로그에 "locked"로 기록되고 있었어요.
나중에 로그를 분석할 때 잠금인지 해제인지 구분이 안 됩니다.
→ "unlocked"로 수정했습니다.

## 다음에 직접 적용할 핵심 3가지

1. **관리자 API는 항상 인증부터** — 내부 서비스라도 엔드포인트별 권한 확인 필수
2. **클라이언트 검증은 UX용, 서버 검증이 실제 방어선** — JS에서 막아도 서버에서 다시 확인
3. **같은 로직은 한 곳에만** — 중복 코드는 한쪽은 방치되고 버그 온상이 됨

## 앞으로 주의할 점 3가지

1. **Rate limiting** — 아직 미적용. `/apply`에 `slowapi`로 IP당 분당 5회 제한 권고
2. **.env 파일 관리** — iCloud 동기화 중이라면 `.env`는 동기화 제외 폴더에 보관하거나 별도 보관
3. **ADMIN_API_KEY 미설정 시 서버 시작 차단** — 현재는 경고만 출력. 운영 환경에서 누락되면 무방비
